# Content Guidelines — AI生成色を消すための執筆規則

Pinterest はクリック後の品質シグナル（直帰率、滞在時間、保存数）を見て将来のリーチを決める。AI臭が強い記事は最初の100クリックで嫌われ、その後表示が止まる。

---

## 禁止フレーズ

これらは AI 生成の典型的な signature。1つでも入っていたら書き直し。

### 英語
- "In conclusion" / "In summary" / "To sum up"
- "Furthermore" / "Moreover" / "Additionally" (段落頭)
- "It's important to note that"
- "When it comes to X"
- "delve into" / "leverage" / "robust" / "seamless" / "ensure" (marketing-speak)
- "In today's digital age"
- "The world of X is constantly evolving"
- "A wide range of options"
- "navigating the complexities of"
- "Explicit weakness:" を全製品の節に等位で並べるのも禁止 — 文章に溶かす
- "Every product has a weakness" / "explicit weakness on every product" — メタクレームは filler
- "In this comparison, only X" を多用しない（1記事1回まで）

### 日本語
- 「〜と言えるでしょう」「〜と考えられます」(常体・敬体ミックスの曖昧表現)
- 「いかがでしたでしょうか」（締めの定型句）
- 「結論から言うと、〜です」(よく使われすぎ)
- 「〜の世界」「〜の魅力」(空虚な総論)
- 「驚くべきことに」「言うまでもなく」
- 「〜することが大切です」(中身のない総論)
- 「ぜひ参考にしてみてください」(常套句)
- **「明確な弱点：」** を全製品節の同じ位置に等位で並べる — 文章に溶け込ませる
- **「全製品に弱点あり」「全製品に明確な弱点あり」** — メタフレーズは禁止
- **「本比較で唯一」「本比較で最も」「本比較で最高」** — 1記事につき本当に意味があるとき1回まで
- **「総評」セクション** で各製品の推薦を逐一再記述する形は禁止。推薦は各製品節で完結させる。締め段落を入れるなら「新しい視点を1つ」だけ、再要約はしない

## タイトルの書き方

タイトル＝検索キーワードの羅列ではなく、見出し。商品名はリードと本文セクションに置く。

- **80文字以内（Unicodeカウント）**
- "A vs B vs C vs D vs E" の連鎖は禁止
- "—" や "：" で区切るサブタイトルは1つまで、コンマ区切りキーワード詰め込みは禁止

### NG例
- `加湿器おすすめ2026年版：5製品を徹底比較 — ダイソン vs パナソニック vs バルミューダ vs Levoit vs Stadler Form、超音波式・気化式・スチーム式の本当の違い、白い粉問題、…` (184字)
- `Best Gaming Mouse 2026: 5 options compared — Logicool G Pro X Superlight 2 vs Razer DeathAdder V3 vs ..., grip style guide, Japan market specifics, explicit weakness on every product` (321字)

### OK例
- `加湿器おすすめ5選 2026 — 白い粉、カビ、メンテの本音比較` (33字)
- `Best gaming mice 2026: sensor accuracy vs DPI hype` (50字)

## 必須要素

### 数字を入れる
- ❌ "Many users prefer NordVPN for its speed."
- ✓ "NordVPN averaged 384 Mbps over my 30-day test, second only to ExpressVPN's 421 Mbps."

### 弱点を1つ書く
完璧な商品はない。各レビュー対象に「これだけは妥協が必要」を1つ含める。

### 1人称の観察
- ❌ "Users will find the interface intuitive."
- ✓ "I needed three minutes to find the kill switch the first time I used it."

### 比較に使う固有名詞を出す
- ❌ "More affordable than competitors"
- ✓ "$3.99/month — half of ExpressVPN's $8.32, and a dollar less than Surfshark"

## 構成

### review (単品レビュー)
1. Hook (1段落): なぜこの製品か、何を達成したかったか
2. Who it's for (200-300語): 対象ユーザ像、合わない人
3. What I liked (300-400語): 具体的な体験 3-4個
4. What's missing (200-300語): 弱点 2-3個
5. Pricing (150-200語): 価格・他社比較
6. Verdict (100-150語): 総合評価、誰におすすめ
7. FAQ (6-10件)

### comparison (比較ランキング)
1. Hook (1段落): 何を解決する記事か
2. How we picked (200語): 評価基準、テスト方法
3-N. 各製品セクション (each 250-400語): #1, #2, ... 順位、スペック、長所、短所
N+1. Side-by-side table (オプション): 価格・速度・機能比較表
N+2. FAQ

### guide (バイヤーズガイド)
1. Hook (1段落): なぜこの記事か
2. What matters most (300-500語): 選定基準を体系的に
3. Common mistakes (200-300語): 初心者がやりがちな失敗
4. Step-by-step (300-500語): 選び方の手順
5. Recommended products (オプション): 各カテゴリ1製品ずつ
6. FAQ

## FAQ の書き方

### NG
Q: 「VPNとは何ですか？」 (一般論はSEOに不要)
A: 「VPNは Virtual Private Network の略で...」(Wikipedia 引用調)

### OK
Q: 「無料VPNと有料VPNはどう違いますか？」
A: 「無料VPNは帯域制限・速度低下・ログ収集（無料の代償としてユーザーデータを売る業者がいる）が一般的です。月$3前後の有料VPNなら速度が10倍以上、データもログされません。」

## 各locale固有のローカライズ

英語記事を翻訳するだけでなく、「その言語圏の読者が知りたいこと」に置き換える。

例: 「ベストVPN」記事の場合
- en: "Best VPN for Netflix US"
- ja: 「日本から海外のNetflixを観るためのVPN」（DAZN・ABEMA等のジオブロック話も追加）
- de: "Bestes VPN für Streaming aus Deutschland"（GEMA・GEZ事情に触れる）
- pt-BR: "Melhor VPN para conteúdos bloqueados no Brasil"（地理的制限の現実話）

## 日本市場サービス記事（レンタルサーバー・SaaS・金融）の追加ルール

もしもアフィリエイト発信のレビュー記事ノウハウ（2026-05 メルマガ「ConoHa WING のレビュー記事で成果を出す5つの鉄則」）を踏まえて、**サービス系商材（rental server / SaaS / クレカ / 証券口座等）** で日本語ロケールを書くときは以下を上乗せする。物販レビュー（VPN以外の家電・コスメ等）は従来ルールのまま。

### spec で `audienceLevel: beginner` を指定したとき

`pipeline/specs/<slug>.yaml` で `audienceLevel: beginner` をセットすると、以下が必須になる:

1. **冒頭セクションに初心者向け前提解説**: 通常の review/guide 構成ではFAQの「〜とは何ですか？」は filler 扱い禁止だが、`beginner` のときに限り **本文セクション (FAQではない)** として「〜とは？」を1つ置いてよい。150-250語、図解1点想定（無くても可）。Wikipedia引用調は依然NG。
2. **安心材料セクション**: review型なら "What I liked" の中ではなく独立した1セクションとして「初心者でも詰まらないポイント」（自動バックアップ／日本語サポート／無料試用期間など）を 200-300語。
3. **申込み手順セクション (Setup walkthrough)**: review/guide型の末尾近くに `subsections` でステップ列挙する形で 3-6ステップ。各ステップに具体的画面名や所要時間（"管理画面 → サイト管理 → ドメイン追加（30秒）"）。スクショ画像はあれば挿入、無ければ言葉で具体化。

### comparison 型での比較表

`comparison` 型では **side-by-side 表** を1つ入れる。Markdown表で書き、列は最大4カラム（料金/速度や容量/サポート/初期費用 など、その商材に合うもの）。表だけで完結させず、表の前後に「読み解き方」段落を必ず置く。

### review/comparison のセット展開

ConoHa WINGのような単品レビューを書いたら、対応する **比較記事**（"レンタルサーバー比較 2026" 的なもの）が registry に既に存在するか確認し、無ければ comparison spec を別途作る。review 記事の末尾段落から比較記事への内部リンクを必ず張る（registry 経由で自動的に拾えるよう slug 関係を `relatedSlugs` で示す方針 — これは未実装、当面は本文中で言及する形でよい）。

### 取り込まないもの（方針衝突）

メルマガで推奨されているが、affiliate_factory 側の方針として **採用しない** もの:

- 「いかがでしたでしょうか」「ぜひ参考にしてみてください」等の締め定型句 → AI生成色のため引き続き禁止
- ランキング順位を ASP 報酬額で決める → ASP-agnostic / 編集独立性のため不可。順位はあくまで実機評価・スペック・価格で決め、報酬は反映しない
- 「迷ったら〇〇」のような結論を最後に1社だけ強調する形 → 1ASP依存リスク。少なくとも2社推奨を残す

## チェックリスト

公開前に：

- [ ] 禁止フレーズ 0件 (`cd site && npm run audit:content` で確認)
- [ ] タイトル ≤ 80 文字（全 locale）
- [ ] `offerNotes` のキーが `meta.ts` の `offerIds` と完全一致
- [ ] 各セクションに具体的な数字が1つ以上
- [ ] 弱点を述べた段落がある（ただし「明確な弱点：」ラベルは使わない）
- [ ] 「総評」「Summary」「Conclusion」セクションは推薦の再要約になっていない
- [ ] FAQ は一般論ではなく実用情報
- [ ] pinDescription はその locale 固有の hook
- [ ] 価格・通貨はその locale 用
